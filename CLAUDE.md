# CLAUDE.md

Project-specific guidance for Claude Code (and other AI assistants) working
in the gents repository.

## What gents is

A Go→TypeScript types generator. Reads Go source, finds structs marked with
`//gents:export`, emits matching TS interfaces plus zero-value factory
functions. Full reference: [docs/gents_package.md](docs/gents_package.md).

## Hard rules

**No third-party dependencies.** Pure stdlib only. No `go get` anything.

**Deterministic output.** Two runs on the same input must produce
byte-identical output. This means:
- Do NOT iterate `map` types in emission paths. The `emitter.marked` map
  is keyed by struct name but is only ever *looked up*, never ranged over
  during emit. Preserve this.
- Source order of structs and fields is preserved through both passes.

**Panic with `file:line` on unsupported input.** Use `emitter.panicAt(pos,
format, args...)`. Never silently skip a field — if gents doesn't know
what to do with a type, the user must see exactly where.

**`encoding/json` is the wire contract.** When in doubt about a type
mapping, ask "what does `json.Marshal` actually put on the wire?" That's
the TS type. See [docs/gents_package.md §3](docs/gents_package.md) for
the full mapping table.

**Never generate a file without the gents header.** The CLI's overwrite
guardrail (`safeWrite`) depends on the exact first line being
`// Code generated by github.com/mmalcek/gents; DO NOT EDIT.`. Don't
add a version suffix, don't rephrase it.

## Adding a type mapping

- **New primitive or `*ast.Ident` form**: extend the switch in
  [emitter.go:mapIdent](emitter.go). Add both the TS type and the factory
  zero value.
- **New qualified selector** (`pkg.Name`): extend the switch in
  [emitter.go:mapSelector](emitter.go). Example: `time.Time`,
  `time.Duration`, `json.RawMessage` live there.
- **New field-level JSON-tag semantic**: add a flag in
  [emitter.go:parseJSONTag](emitter.go), then pipe the flag through
  `collectFields` into a coercion helper like `applyStringFlag`. The
  `,string` handling is the template.
- **Third-party types at runtime**: these are the user's problem via
  `-map` / `Options.TypeMap` — don't hardcode anything package-specific.

## Adding a golden fixture

1. `testdata/<name>/input.go` — a complete Go file, `package <name>` (no
   spaces/dashes; match the dir name).
2. `testdata/<name>/expected.ts` — the exact TS output, one trailing
   newline.
3. Add an entry to `goldenCases` in [emitter_test.go](emitter_test.go).
4. Run `go test -run TestGenerateMatchesGolden/<name> ./...` to confirm.
5. `TestIdempotent` runs the same fixture 10× and compares — no
   additional work needed, it picks up new entries automatically.

## Adding a panic fixture

1. `testdata/panic_<thing>/input.go` — triggers the panic condition.
2. Add an entry to `panicCases` with a substring the panic message must
   contain. Use a stable, specific phrase (e.g. "channel types", not
   "error").
3. The test framework asserts both the substring and the presence of a
   `file:line` pointer in the panic message.

## Bundle-mode fixtures

Use `testdata/<name>/input/` (directory, not a single file). Register
with `isDir: true` in `goldenCases` or `panicCases`. The runner calls
`gents.GenerateDir` instead of `gents.Generate`.

## Testing workflow

- Primary gate: `go test -count=1 ./...`
- Race check (matches CI): `go test -count=1 -race ./...`
- Vet: `go vet ./...`
- CI runs all three on every push to main and every PR. Keep green.

When a golden test fails, the framework prints a diff-style side-by-side
(`--- got ---` / `--- want ---`). Read the difference, then either fix
the emitter or update the fixture if the change is intentional.

## Things NOT to do

- Add a third-party dependency (see "Hard rules").
- Iterate over a `map` to drive emission order.
- Introduce a `time.Time`→`Date` default or anything else that embeds a
  project-specific opinion. `-map` exists for that.
- Weaken the overwrite guardrail in `safeWrite` — no "best effort" bypass.
- Remove the `file:line` pointer from any panic — it's load-bearing for
  UX.
- Break `omitempty` / `omitzero` / `,string` semantics. They match
  `encoding/json` exactly; don't silently diverge.
- Rename or reformat the `// Code generated by ...` header.

## When you need context

- **User asks "why does X map to Y?"**: `json.Marshal(X)` and compare.
  If the shapes match, gents is correct.
- **User asks for a new feature**: check §7 Limitations & non-goals in
  the reference doc before implementing — some things are explicitly
  out of scope.
- **You hit a design decision point that's not covered here**: check
  the git log on the relevant file. Design reasoning often lives in
  commit messages.
